# websudoku

Personal recreation of [websudoku.com](https://www.websudoku.com/) for Roger's own use — the original UI he's used to, plus the customizations the real site never offered. Frontend only; not published publicly (repo is private: it contains the original site's logo and CSS-derived styling).

## Stack (deliberate choices — don't change without asking)

- **Vite + vanilla JavaScript.** No TypeScript, no framework — explicitly chosen over React/Next as overkill.
- Puzzles from **qqwing 1.3.4** (client-side, no backend) via `src/generator.js` — a real generator with technique-based grading, replacing the earlier sudoku-gen seed-bank. Puzzle numbers are REAL: the number seeds a mulberry32 PRNG swapped into `Math.random` for the whole generate-and-grade loop, so "Select a puzzle 12,345" always deals the same puzzle. **Numbering contract: the qqwing version (pinned, no caret), the seed hash, and the acceptance bands in `generator.js` all determine which puzzle a number produces — changing any of them renumbers every puzzle.** Difficulty bands were calibrated by grading live websudoku.com puzzles with qqwing's technique counters, then re-tuned July 2026 against 12 live puzzles/level to sit a notch *above* the real site's hardness on Hard (Roger's request) — that re-tune renumbered Hard and Evil (Easy = singles-only @ 34-36 givens, padded up from qqwing's natural 28-29; Medium = singles/hidden singles @ 28-31; Hard @ 26-28 = per-number seeded 40% chance of demanding pairs (2-6 steps, live site is ~1/3 @ ~2.4 steps), else hidden-single-heavy singles-only). **Evil was rebuilt again July 2026 to NYT-Hard structure (renumbering Evil a second time)**, calibrated against 1,155 archived NYT Hard puzzles: generated with `Symmetry.NONE` (the one asymmetric level) because individual-cell digging makes every grid **minimal** — no removable given (symmetric pair-digging left ~2-3 redundant givens, the padding that made Evil feel looser than NYT); acceptance = qqwing Intermediate, ≤26 clues (deals 23-26, mean ~24.6), ≥3 pair steps with **no upper cap** (mean ~5.3, tail to 10+ for NYT-style variety; the floor kills live-websudoku's 30% one-pair-step duds), and **opening width ≤4** (cells findable via naked/hidden single on the raw grid — matches NYT's 3.6 tight-opening mean and steepens the endgame cascade to ~2×). `openingWidth()` in `generator.js` is a pure function of the grid (consumes no seeded-rng draws) and is short-circuited last in the acceptance test. Evil deals accept ~1 in 50 (~0.4s in Node, ~1.5-2s on the phone). NYT Hard's remaining edge — ~7% of their puzzles need beyond-pairs techniques (XY-wing/chains) — is deliberately NOT replicated: qqwing's Expert tier (guessing required) stays unused at every level, including Beyond Evil, reserving beyond-pairs for a possible future harder level (asymmetric qqwing deals hit Expert ~35% of the time, so it'd generate cheaply). Easy/Medium/Hard/Beyond remain 180°-rotationally symmetric with unique solutions, like the original's; analysis notes live in `docs/nyt-evil-notes.md`. **Known, accepted divergence: every Evil opens with exactly one digit absent from the givens** (greedy digging always strips one digit bare; NYT guards this, 5% there) — Roger reviewed it 2026-07-12 and kept it (play-texture metrics are at parity; it's cosmetic). A faster coverage-aware custom digger was prototyped and declined — do NOT "fix" this without asking; details + prototype numbers in the notes doc. qqwing is GPL-2.0 — fine while private.
- **vite-plugin-pwa** for offline/installable PWA (planned: deploy to Roger's personal Vercel, install on iPhone). `workbox.globPatterns` includes `png` so the logo/icons precache — keep that.

## Fidelity principle

Visual fidelity to the original site matters. The markup and CSS deliberately mirror the real websudoku page (Roger pasted its inner-frame HTML during development):

- Grid cells: `<td class="{c,e,f,g,h,i}{0-3}">` — letter picks which borders are thick 3×3 box borders (e/g/i = thick left, f/g = thick top, h/i = thick bottom), digit 0–3 is the escalating red error level. Ids `c{col}{row}`, inputs `f{col}{row}` (column-first, like the original).
- Inputs: `s*` = givens (Times), `d*` = user entries (lucida handwriting → Comic Sans MS fallback, blue), `w*`/`v*` = the original's inline pencil-mark styles (no longer applied, kept in CSS for reference).
- Original layout: full-height table, grid vertically centered, message line in `bigbig` above the grid ("Here is the puzzle. Good luck!" etc. — texts lifted from the original's JS vars), level links in the left column, buttons `How am I doing? / Pause / Clear / Options...` (Print button was removed on request), "Hard Puzzle N - 0:42 - Select a puzzle..." info line below the grid.
- All colors go through CSS variables in `:root` / `html[data-theme="…"]` blocks so themes are a variable swap. Original hex values live in the light theme (`:root`).
- Deliberately omitted: ads, analytics, accounts, cookie banner, Variations/Ebook/Deluxe links.

## Custom behavior (differs from the original on purpose)

- **Data model**: `game.entries[81]` (committed single digits) is separate from `game.marks[81]` (pencil candidates). Entry and marks are mutually exclusive per cell; setting one clears the other.
- **Pencil marks render as a 3×3 mini-grid** (`.pm` overlay div per cell, digit d in fixed slot d) — modern-app style, not the original's digits-in-a-row.
- **Keyboard**: Arrows/WASD move; 1–9 enters (pressing a cell's current digit again clears it — everything is a toggle); hold **Space or Shift + digit** toggles a pencil mark; Backspace/Delete clears the cell; Enter = check; **Escape hides the selection box**, and any movement key re-summons it where it was (center cell on a fresh puzzle). Typing is handled in `keydown` (inputs are `maxLength=1`, `inputmode="none"`).
- **Touch controls** (`settings.touchControls`: `'thumbpad'` | `'keypad'` | `'none'`, Options select; default `thumbpad` on coarse pointers, `none` otherwise). Old boolean `showKeypad` settings migrate: keypad-on-touch installs land on `thumbpad` (the new phone default), keypad-on-desktop keeps `keypad`. `html.thumbpad` shows the thumb pad + its phone bottom clearance; `html.no-keypad` (set whenever the mode isn't `keypad`) removes the keypad's clearance/#focus-exit offset. On phone portrait (≤480px) the grid goes full-width via fluid `--cell` sizing (`min(44px, (100vw - 24px)/9)`) so the board fits at any zoom/Display Zoom.
- **Thumb pad** (`src/thumbpad.js`, `#thumbpad`, fixed bottom-right, game-only, ~`min(46vw, 200px)` wide; **shown only under `(pointer: coarse)`** — a desktop with the setting stored stays clean, and its phone clearance is likewise gated to coarse+≤480px; **board-only mode uses a small clearance (36px + safe-area)** that floats the centered grid a slight gap above the widget): two panels in one **identical footprint** — the digit pad packs 4 equal rows (3 digit + actions) into the map's height, so swapping panels never grows the widget. **Mini-map** = live 9×9 miniature of the board (Times givens, handwriting-colored entries, `·` for mark-only cells, error tint, `.sel` ring; box lines are a four-gradient `::after` overlay). **Absolute trackpad aiming**: pointerdown/drag on the map highlights the matching REAL-board cell (`.t td.aim` + mini `.aim`), lift selects it (focuses the input — the focus handler calls `thumb.setSelected`) and flips to the **digit pad** (givens: select only). Digit pad = 3×3 digits laid out like the pencil mini-grid: tap toggles the entry then auto-returns to the map; **long-press (500 ms) toggles a pencil mark and STAYS** so marks chain; ▦ = back, ⌫ = clear+return; `.active` = cell's entry, dot = penciled. Taps bypass the keypad's `pencilMode` — `thumbTarget()` in main.js guards (done/given/no-selection) and re-focuses like `applyKey`. main.js owns all state: `paint()` calls `thumb.paintCell(i)`, deals/restores call `thumb.reset()`; `hooks.getCell` is the only read path.
- **Side keypad** (`touchControls = 'keypad'`; 1–9, ⌫, ✎ pencil toggle) — the original's control, kept as an option. **On phone portrait (≤480px) it docks as a fixed bottom thumb bar** (1–9 row, then ⌫/✎; safe-area aware). **Long-press a digit key toggles a pencil mark** — the touch analog of Space/Shift+digit; the trailing click is swallowed via a `longPressed` flag. `applyKey` re-focuses the remembered `selected` cell if focus was dropped (iOS focus is fragile), so the keypad never dead-ends.
- **Thumb pad** (`touchControls = 'thumbpad'`, the touch default; `src/thumbpad.js`, state owned by `main.js` via hooks): mini-map + 3×3 digit pad sharing one bottom-right footprint (phone portrait + coarse pointer only). Map = absolute trackpad: dragging aims (`.aim` on the real board's td; the grid table also gets `.aiming`, which CSS uses to hide the *selection's* `:focus-within` box so only one cursor shows), lifting selects — a given selects only, an open cell flips to the digit pad. Pad: tap = entry toggle + back to the map, long-press = pencil mark + stay. **The real board drives the pad too: tapping an editable cell opens it, tapping a given — or any touch outside board+pad (document-level `pointerup`) — drops back to the map.** On coarse pointers the grid's focus handler skips `input.select()` (iOS renders it as a blue text-selection smear on the digit); grid inputs are additionally `user-select:none` (WebKit's UA sheet makes inputs selectable despite the `.t` rule) and the grid/pads set `-webkit-tap-highlight-color:transparent`.
- **Zoom hardening**: `user-scalable=no, maximum-scale=1` (honored in the standalone PWA; Safari-in-browser ignores it), `touch-action: manipulation` kills double-tap zoom, dialog inputs are ≥16px on coarse pointers so iOS never auto-zooms on focus, `viewport-fit=cover` + `env(safe-area-inset-*)` padding for notch/home-indicator. `theme-color` meta is updated from JS to match the current theme's page background. **`apple-mobile-web-app-status-bar-style: black-translucent`** extends the standalone PWA's viewport to the physical screen top so the ambient video reaches under the Dynamic Island (Roger's request); accepted cost: iOS renders status-bar glyphs white on every theme (washy on light day themes; not switchable at runtime). Layout compensates via `max(…, env(safe-area-inset-top))` on `#menu`/`#right-column`/`#focus-exit`; a change to this meta may need the PWA removed and re-added to take effect.
- Options modal: theme (`settings.theme`: **auto** + 6 light themes (light/sage/sky/sepia/lavender/blush) + 4 dark (dark/midnight/forest/ember) — auto follows the OS live via a `prefers-color-scheme` change listener, flipping between a chosen day/night theme pair (`settings.autoLight`/`autoDark`, classic light/dark by default; sub-selects visible only while theme = Auto); unknown/removed theme keys fall back to auto; `DARK_THEMES` in `main.js` must stay in sync with the two `:is()` dark-family selector lists in `style.css` (shared dark vars + logo invert); the `theme-color` meta reads the computed `--page-bg` after the swap; palettes are research-derived and WCAG-checked — rationale + citations in `docs/theme-notes.md`; ember is the warm night theme, the only one whose *entries* aren't blue (amber, melanopic-light rationale); old boolean `darkTheme` settings are migrated), timer, allow pencil marks, highlight-on-check vs purple-count message (mirrors the original's two check modes), check-as-you-type, touch controls (thumb pad / keypad / none; the keypad-pencil sub-toggle only shows for keypad). Settings/stats/current game persist in localStorage (`websudoku:settings|stats|game`); saved games restore across reloads and old formats are migrated.
- **Beyond Evil** (5th level, key `beyond`): an Evil that additionally forces a *hidden pair in a row/column within the first 3 placements* — the "opening wall" Roger hit on live websudoku and wanted on demand. Acceptance is ~1 in 2,500 qqwing candidates, far too slow to generate live, so it deals from a **pre-generated bank** (`src/beyond-bank.js`, made by `scripts/make-beyond-bank.mjs` with fixed seeds — rerunning reproduces the identical bank while qqwing stays pinned). **Bank numbering contract: the bank ORDER is the numbering ("Beyond Evil Puzzle 7" = entry 7) — never reorder or insert, append only.** Numbers wrap via mod, `maxPuzzleNumber('beyond')` = `BEYOND_COUNT` in `generator.js` (keep it equal to the bank length). The bank is a dynamic-import chunk so it loads on first Beyond deal and precaches for offline; solutions aren't stored — qqwing re-solves client-side (instant, unique).
- **"Select a puzzle..."** in the info line opens a number-entry dialog; the number deterministically reproduces that puzzle at the current level (saved games from the sudoku-gen era restore fine but their numbers predate the contract and won't re-deal the same grid).
- **Ambient background** (`settings.ambient`, Options checkbox, off by default; game-only — never the menu): fixed `#ambient` layer at `z-index:-1` with a `color-mix` page-bg scrim (55%) so chrome text stays readable — **sized `100vw`/`100vh`, NOT `inset:0`**: with iOS Safari Page Zoom (a user setting that carries into the standalone PWA, diagnosed live on Roger's phone at 150%), the zoomed layout viewport ends ~a status bar short of the physical screen bottom, so `inset:0` left a page-bg strip there; `vh` resolves against the full screen. Default renderer is `src/ambient.js`, a generative parallax "driving past the city" canvas — palette derived from the theme's CSS vars (`--page-bg`/`--box-border`/`--glint`; page-bg luminance < 0.5 = night mode with lit windows/stars/headlight streaks, else day with clouds), ~30fps cap, stops when hidden/menu, honors `prefers-reduced-motion`, uses its own mulberry32 (Math.random is off-limits — the generator swaps it). Dropping a video at `public/ambient.mp4` auto-overrides the canvas (detected via `canplay`/`error`); the SW runtime-caches it (CacheFirst + rangeRequests in `vite.config.js`) and `ambient.js` primes that cache with a full fetch because `<video>` range requests (206) aren't cacheable. A YouTube link (Options field, `settings.ambientYouTube`, stored as the bare 11-char id via `parseYouTubeId`) outranks both **while online**: muted nocookie embed, `scale(1.3)` crop hides branding, `pointer-events:none`, play/pause via `enablejsapi` postMessage; offline/reduced-motion drops back to mp4/canvas automatically (accepted tradeoffs: possible ads, brief branding flash, network-only).
- **"Just the puzzle" mode** (`settings.boardOnly` → `html.board-only`): hides everything except the grid and (on touch) the keypad (the message line too, so the board centers true; Enter-check feedback is invisible in the mode), and locks scrolling (`overflow:hidden` + `overscroll-behavior:none`) so phones get a fixed, non-scrolling solving surface. Enter via the "Just the puzzle" link next to the theme link, the **F** key, or (touch) **double-tapping empty background** — a manual two-tap detector on `pointerup` in `main.js` (350ms/48px, `pointerType === 'touch'` only; iOS doesn't fire `dblclick` for touch), which excludes the grid/keypad/thumbpad/links/inputs and toggles both ways; exit via the corner ✕, F, or the same double-tap. **iOS pan guard**: iOS scrolls to reveal a focused input even inside `overflow:hidden` boxes — including the body element, which `window.scrollTo` never resets and whose scroll events don't bubble — and never scrolls back (page + fixed ambient layer stuck shifted up, page-bg bar at the bottom). Defenses: **`html.board-only body {position:fixed}`** (a fixed body has nothing to displace — the load-bearing one), all grid `focus()` calls pass `{preventScroll:true}` (native tap-focus can't be intercepted, hence the CSS), entering the mode resets window + body scroll, and a capture-phase document scroll listener zeroes any element iOS still displaces while the mode is on. Persisted, so the iPhone PWA can live in it. Pause/check/level-switching require exiting the mode (win detection still fires automatically).

## Glyph centering — DO NOT "simplify" these rules

The `transform: translateY(...)` rules on `.s*`, `.d*`, `.pm span` and the `@supports (-moz-appearance: none)` Gecko override in `src/style.css` were **empirically calibrated** by rendering test pages in headless Chrome *and* headless Zen (Roger's browser — a Firefox fork; Gecko draws Times ~0.09em high in inputs, Blink ~0.02em, and the handwriting font sits ~0.04em low in both). Method: screenshot a magnified given/entry pair, decode the PNG in Node, measure ink centers against the cell borders. Result is within ±0.25 CSS px in both engines at all breakpoints. The original site has the misalignment; this clone fixes it. Inputs fill cells via `position:absolute; inset:0` (percentage heights inside table cells don't resolve in Gecko). WebKit/iOS has never been calibrated — if digits look off on the iPhone PWA, redo the measurement loop against Safari.
